## 2. 项目定位与核心需求

## 2.1 项目定位

- EzClipboard 是一个基于 `uTools` 的本地剪贴板管理插件，不是普通 Web 站点。
- 运行模式是：
  - `uTools` 主窗口
  - 前端 Vue 页面
  - `preload` 注入宿主能力
  - 本地剪贴板监听程序
  - 本地数据库 / 宿主数据库存储
- 相关基线说明见 [`docs/VersionDesc/20260331-v0-当前版本架构与迭代基线说明.md`](../../../VersionDesc/20260331-v0-%E5%BD%93%E5%89%8D%E7%89%88%E6%9C%AC%E6%9E%B6%E6%9E%84%E4%B8%8E%E8%BF%AD%E4%BB%A3%E5%9F%BA%E7%BA%BF%E8%AF%B4%E6%98%8E.md#L1)。

## 2.2 核心业务需求

- 自动监听系统剪贴板变化，识别文本、图片、文件并入库。
- 提供主列表浏览、搜索、筛选、收藏、删除、锁定、复制、粘贴等核心能力。
- 允许收藏数据独立管理，支持标签和备注。
- 支持较强的键盘操作：
  - 单步导航
  - 分页导航
  - 快捷复制/粘贴
  - 多选操作
  - 抽屉、详情、清理弹窗等层级切换
- 作为桌面工具，重点是：
  - 快
  - 稳
  - 键盘友好
  - 长列表不乱

## 2.3 当前重点需求

- 当前正在推进的重点不是“新增一个独立业务模块”，而是：
  - 提升主列表交互性能
  - 修复单步/分页/长按导航问题
  - 修复删除后高亮错位和闪烁
  - 修复锁/收藏状态刷新与行头布局问题
  - 在必要时升级工程底座，使后续优化更可持续

## 4. 运行时与宿主桥接

## 4.1 preload 注入

- [`public/preload.js`](../../../../public/preload.js#L1) 把以下能力注入 `window.exports`：
  - `utools`
  - 文件系统能力
  - `listener`
  - Electron `clipboard` / `nativeImage`
  - `time`
  - `Buffer`

## 4.2 前端依赖模式

- 当前前端大量依赖全局对象：
  - `window.exports`
  - `window.db`
  - `window.listener`
  - `window.remove`
- 这意味着项目不是严格依赖注入式架构，而是宿主桥接型架构。

## 4.3 运行时约束

- 任何现代化重构都必须继续兼容：
  - `uTools` 主窗口静态入口
  - `preload.js`
  - 插件环境中的全局能力注入
- 不能把项目误当成纯浏览器 SPA 去重构。

## 5. 数据存储结构

## 5.1 当前主存储

- 当前基线文档说明项目历史上主要使用本地 JSON 文件存储，见 [`docs/VersionDesc/20260331-v0-当前版本架构与迭代基线说明.md`](../../../VersionDesc/20260331-v0-%E5%BD%93%E5%89%8D%E7%89%88%E6%9C%AC%E6%9E%B6%E6%9E%84%E4%B8%8E%E8%BF%AD%E4%BB%A3%E5%9F%BA%E7%BA%BF%E8%AF%B4%E6%98%8E.md#L88)。
- 当前 `initPlugin.js` 中仍定义了本地 DB 结构与初始化逻辑，见 [`src/global/initPlugin.js`](../../../../src/global/initPlugin.js#L96)。

## 5.2 数据结构

- 当前数据库主结构：
  - `data`: 普通历史记录
  - `collects`: 收藏 ID 列表
  - `collectData`: 收藏完整数据
  - `tags`: 全局标签
  - `tagUsage`: 标签使用统计
  - `createTime`
  - `updateTime`

## 5.3 收藏模型

- 收藏不是简单 `collect: true` 标记，而是“普通历史”和“收藏数据”分离存储。
- 收藏逻辑要点：
  - 收藏时从普通历史迁出，插入收藏集
  - 取消收藏时从收藏集移回普通历史
  - 收藏项可携带 `collectTime`、`tags`、`remark`
- 这会影响：
  - 删除逻辑
  - 搜索逻辑
  - 列表混排逻辑
  - 高亮恢复逻辑

## 5.4 迁移与兼容

- [`src/global/initPlugin.js`](../../../../src/global/initPlugin.js#L116) 内有多段兼容历史数据的规范化逻辑，包括：
  - `collect` 旧字段迁移
  - `collectData` 修复
  - `collectTime` 补齐
  - `tags` / `remark` 补齐
  - `locked` 补齐
  - `originPaths` 补齐
- 说明：
  - 当前项目有真实历史包袱
  - 存储层改动不能想当然清空旧兼容逻辑

## 5.5 备用 / 新方向存储

- 仓库中已有 [`src/global/utoolsDB.js`](../../../../src/global/utoolsDB.js#L1)，说明项目已经尝试或规划使用 `uTools DB API` 替代 JSON 文件。
- 该实现特征：
  - 以文档 `_id` 做命名空间管理
  - 把历史项、收藏、标签拆成多个文档存储
  - 支持批量写入和清空迁移
- 这意味着后续 AI 评估时需要考虑：
  - 是否继续沿用 JSON 文件
  - 是否切换到 `uTools DB`
  - 是否保持双轨兼容

## 6. 剪贴板监听与入库链路

## 6.1 监听程序

- [`public/listener.js`](../../../../public/listener.js#L1) 会按平台启动原生监听程序：
  - Windows `.exe`
  - Linux 可执行文件
  - macOS 可执行文件
- 监听到 `CLIPBOARD_CHANGE` 后，会向前端宿主发事件。

## 6.2 入库流程

- 入口在 [`src/global/initPlugin.js`](../../../../src/global/initPlugin.js#L1)。
- 基本链路是：
  1. 初始化宿主能力与数据库
  2. 启动监听
  3. 读取剪贴板
  4. 识别文本 / 图片 / 文件
  5. 转成统一 item
  6. 写入数据库
  7. 推动前端 UI 刷新

## 6.3 相关风险

- 监听失败时有回退逻辑。
- 因为入库链路与 UI 刷新耦合较近，如果重构 UI 层，要避免拿页面层去补监听层问题。

## 9. 当前需求重点与评估焦点

## 9.1 功能性需求

- 单步导航：完全可见则不滚。
- 长按导航：统一半分页/分页响应。
- 分页导航：与长按行为一致。
- 删除恢复：高亮稳定，不闪烁、不选错。
- 锁/收藏状态：及时刷新。
- 混排搜索 / 收藏块 / 锁筛选：继续可用。

## 9.2 性能需求

- `300ms` 定义为：
  - 单次交互后 UI 达到滚动稳定且高亮正确的完成时间。
- 核心不是跑分，而是：
  - 降低无意义滚动
  - 降低重渲染范围
  - 降低状态竞争

## 9.3 评估时必须关注的边界

- `uTools` 插件运行环境
- `preload.js` 兼容
- 收藏数据分离模型
- 多选、预览、懒加载与键盘体系耦合
- 插件环境与浏览器开发环境差异

## 10. 已确定迁移方案：S2

## 10.1 S2 定义

- 工程底座：
  - `Vue 3.5 + Vite`
- 列表方案：
  - `TanStack Virtual`
- 重构范围：
  - 主列表模块重写
  - 导航状态层重写
  - 滚动策略层重写
  - 保持 `uTools` 插件运行方式

## 10.2 选择 S2 的原因

- 当前问题根因主要在列表交互层，不是单纯旧框架慢。
- 如果只升级构建器，不重写列表内核，收益有限。
- `TanStack Virtual` 更适合高定制键盘交互和 headless 控制。
- `Vite` 更适合作为新的工程基线，能降低后续维护成本。

## 10.3 S2 目标结构

- 构建层：
  - 用 `Vite` 取代 `Vue CLI + webpack 4`
- 列表层：
  - 导航状态层
  - 滚动策略层
  - 行渲染层
- 页面层：
  - `Main.vue` 继续负责 `currentShowList` 真相源和页面级状态编排

## 10.4 S2 不改变的内容

- 不改变 `uTools` 插件形态。
- 不改变数据库业务语义。
- 不改变收藏独立存储模型。
- 不改变主业务能力和主要快捷键语义。

## 11. 对 AI 评估者的建议问题

- 如果按 S2 重构：
  - 是否建议继续保留现有 `Main.vue` 作为页面级状态中心？
  - 是否建议把导航状态提到独立 hook？
  - 是否建议先迁移构建，再重写列表，还是并行推进？
  - 是否建议同时切换到 `uTools DB`？
  - TanStack Virtual 的 item measurement、scrollToIndex、overscan 策略应该如何设计？
- 如果不按 S2：
  - 是否有比 `Vue 3.5 + Vite + TanStack Virtual` 更适合 `uTools` 插件的现代组合？
  - 如果有，切换收益是否显著超过迁移成本？

## 12. 建议结论

- 对不同 AI 的评估任务，建议至少输出以下内容：
  - 是否认同 S2 是当前最优主线
  - 若不认同，替代方案是什么
  - 对本项目的存储层是否需要同步重构
  - 对 `Main.vue` / `ClipItemList.vue` / `hotkeyRegistry` 的重构边界建议
  - 风险最高的 3 个点和对应缓解策略
