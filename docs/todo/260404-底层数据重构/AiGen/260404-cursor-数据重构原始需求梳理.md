## 一、ClipboardShelve 数据处理和加载逻辑（底层实现）

### 1.1 数据库结构（内存 + 单文件 JSON）

核心形态（`DB.defaultDB`）：

- `data`: 普通历史记录数组  
- `collects`: 收藏项 id 列表  
- `collectData`: 收藏项完整数据（与历史链分离，避免被 maxsize/过期策略误删）  
- `tags` / `tagUsage`: 标签及使用次数  
- `createTime` / `updateTime`: 库级时间戳  

持久化：设备维度路径（`getNativeId()` + `readSetting` 中的 path 配置），`writeFileSync` 全量 `JSON.stringify` 写回。

**代码锚点**: `src/global/initPlugin.js` 中 `class DB` 构造与 `defaultDB`（约 L111–L127）、`updateDataBaseLocal`（约 L302–L310）。

### 1.2 初始化与迁移

1. 若文件存在：`readFileSync` → `JSON.parse`  
2. **迁移**: `collects` 从旧 `item.collect` 拉出；`collectData` 从 `data` 按 `collects` 补全；补全 `collectTime`、`tags`、`remark`、`locked`、`originPaths`  
3. **过期**: `setting.database.maxage != null` 时按天裁剪 `data`（收藏 id 不参与误删逻辑，与 collect 集合对齐）  
4. **一致性**: `collects` 与 `collectData` 互滤，去除幽灵 id  
5. 新建库：`defaultDB` + `updateDataBaseLocal`  

**锚点**: `src/global/initPlugin.js` `DB.init()`（约 L128–L273）。

### 1.3 多窗口 / 外部写入同步

`fs.watch(this.path)`，`change` 时重新读文件、`JSON.parse`，更新 `this.dataBase` 与 `window.db.dataBase`，并 `listener.emit('view-change')`。

**锚点**: `watchDataBaseUpdate`（约 L274–L292）。

### 1.4 剪贴板获取与监听

- **`pbpaste`**: 优先文件（`utools.getCopyedFiles`）；过滤「全部来自插件临时目录」的假 file；再 text；再 image（`clipboard.readImage` + `toDataURL`）。Shelve 额外包含源路径、活动窗口信息等字段。  
- **变更处理** `handleClipboardChange`: md5(id) 来自 `item.data`；已存在则 `updateItemViaId`；否则 `addItem`。Shelve 含 **写回剪贴板** `restoreClipboard`、`isRestoringClipboard` / `lastRestored*` / 次数上限等防循环。  
- **监听**: 原生 `listener` 的 `change/close/exit/error`；失败或不可用时 **轮询降级**（`time.sleep(300)` + `pbpaste` + id 比对），并防重复注册。

**锚点**: `pbpaste`（约 L886–L951）、`restoreClipboard` / `handleClipboardChange`（约 L978–L1070）、`addCommonListener` / `registerClipEvent`（约 L1072–L1146）。

### 1.5 增删改与收藏、标签

- **addItem**: `unshift` 到 `data`，空文本跳过，`maxsize` 时从尾部弹非收藏；结束时 **`updateDataBaseLocal()`**（同步写盘）。  
- **收藏/取消收藏、标签 CRUD、排序**等：多处直接 `updateDataBaseLocal()`，部分路径走 **`queuePersist()`**（`updateDataBase` + **300ms 防抖**再 `updateDataBaseLocal`）。  
- 对外暴露：`window.db`、`window.remove`、`window.queuePersistDb` 等（文件末尾）。

**锚点**: `addItem`（约 L311–L343）、`queuePersist`（约 L297–L300）；收藏与标签逻辑在同文件后续大段（Cascade 文中 L440–L776 区间，以当前仓库为准）。

### 1.6 视图层：列表数据如何从 DB 到 UI

- `Main.vue`：`list` / `showList`、`GAP=15` 懒加载增量、`parseStarFilter`（`*` + 标签/正文）、`updateShowList`、`getItemsByTab`，过滤链多次 `filter`。  
- `ClipItemList.vue` 等与 `v-memo`、Teleport 预览相关（具体以组件为准）。

**锚点**: `Main.vue` 中 `updateShowList`（约 L609–L657）、`getItemsByTab`（约 L659–L668）；批量清理 `clearRegularTabItems` / `clearCollectTabItems`（约 L683–L721，`forEach` + 逐条 `remove` / `removeCollect`）。

### 1.7 快捷键：与数据层的衔接方式

- **配置**: `hotkeyBindings.js` 中 `HOTKEY_BINDINGS` + `getEffectiveBindings()`（`utools.dbStorage` 的 `setting.hotkeyOverrides`）。  
- **分发**: `hotkeyRegistry.js` — `registerFeature(featureId, handler)` + `setBindings` + 按 layer/state 解析 shortcut，**featureId 串联 UI 行为**。业务效果仍分散在 `Main.vue`、`useClipOperate.js` 等，**不是**统一的「领域 Action + 快捷键只挂 ActionId」的一层模型。

**锚点**: `src/global/hotkeyBindings.js`、`src/global/hotkeyRegistry.js`。

### 1.8 对 Cascade 稿的复核修正（融合时必填）

| 点 | 说明 |
|----|------|
| 防抖写入 | Shelve **仅有** `queuePersist()` 路径使用 300ms 防抖；**`addItem`、多处 remove/sort/collect 仍直接 `updateDataBaseLocal()`**，与「所有写入都防抖」不同。 |
| ClipboardManager 对比行号 | Manager 的 `initPlugin` 独立于 Shelve；链接应以姊妹仓库 `ClipboardManager/src/global/initPlugin.js` 为准（本工作区两 repo 并列时路径为同级目录）。 |
| utools.db 示例 | 官方本地库单文档约 **1MB** 限制；全库大图若整体 `put` 进单 document 可能不适用，需拆 doc 或外链文件（见第三节）。 |

## 二、与 ClipboardManager 的实现对比

二者同源架构：**渲染进程**内 `initPlugin` + **preload** 注入 Node/Electron API；数据均为 **单 JSON 文件 + `writeFileSync`** + **`fs.watch` 同步视图**。

| 维度 | ClipboardShelve | ClipboardManager |
|------|-----------------|------------------|
| 默认数据结构 | `data` + `collects` + `collectData` + `tags` + `tagUsage` | 仅 `data`（收藏为 `item.collect` 布尔） |
| init 时过期 | `maxage == null` 时不按时间删 | 按 `setting.database.maxage` 过滤（需结合 Manager 的 readSetting） |
| 写入防抖 | 部分操作 `queuePersist` 防抖 | 无防抖，多路径直接 `updateDataBaseLocal` |
| 剪贴板 file | 忽略「全来自插件临时目录」的 file | 不区分临时目录 |
| 源窗口/路径 | text/image/file 可带 `sourcePaths`、窗口标题等 | 无 |
| 写回剪贴板防清空 | `restoreClipboard` + 防循环 | 无 |
| 监听降级 | 事件失败降级轮询 + 防重复注册 | `error` 时 `addCommonListener`，`close/exit` 直接 `outPlugin` |
| UI 列表 | 懒加载 GAP、`*` 收藏搜索块、`v-memo` 等 | 更简（无标签子系统等） |
| 快捷键 | 分层 + feature 注册 + dbStorage 覆盖 | 无对等级别 |

**Manager 锚点**: `ClipboardManager/src/global/initPlugin.js`（如 `DB` L16–L138、`pbpaste` L141–L162、监听 L196–L234）。

## 三、uTools 框架特点与高效实现方向（检索摘要）

以下综合公开文档与常见插件实践（具体以 [uTools 开发者文档](https://www.u-tools.cn/docs/developer/docs.html) 为准）：

1. **preload**  
   - 在 `plugin.net` 文档中说明：可访问 Node 能力；**勿混淆打包/压缩 preload**（须保持可读）。  
   - 与渲染页通过挂载 `window` 桥接（与本项目 `window.exports` 一致）。

2. **存储**  
   - **dbStorage**：键值、偏配置与小数据。  
   - **本地数据库（utools.db）**：文档型、离线优先、同步能力；**单条文档有大小上限**（文档载明约 1MB），适合 **按条目标 id 分片** 或小字段 + 外链文件。  
   - 与本项目当前「整库一个 JSON」相比：分片后可 **减少单次序列化体积**、便于按 id 增量更新（需自管索引与迁移）。

3. **剪贴板与性能**  
   - 仍在 preload/主进程侧读剪贴板；大图 `readImage` + `toDataURL` 在主路径上 **易阻塞**。方向：**磁盘存 blob/文件路径、列表只保留缩略图或延迟解码**。  

4. **Web Worker**  
   - Worker 不能任意访问 `utools`；重计算宜在 **preload 子进程**、或 Worker 内纯 JS 处理（如压缩、hash），IO 仍回传主线程/preload。  

5. **与「最高效」相关的结论性建议（不限于当前实现）**  
   - **数据**：历史项按 id 分文档，或使用带索引的存储，避免每次全量 `JSON.stringify`。  
   - **大图**：内容落盘，DB 仅存路径 + 元数据。  
   - **列表**：虚拟列表 + 可视区装载图片/全文。  
   - **批量操作**：批量合并为少量写盘、或异步队列 + `requestIdleCallback`/分片，`updateShowList` 防抖。  
   - **快捷键**：领域 **Action**（纯函数/服务层，`copyItem(id)` 等）与 **绑定表**（shortcut → actionId + payload）分离，与现有 `featureId` 注册可渐进映射。

## 四、原始痛点与现状对照

| 痛点 | 现状要点 |
|------|-----------|
| **itemList 渲染** | 过滤对 `getItemsByTab` 全量或大块 slice；多条 filter；图片大数据在 item 上；懒加载只减轻 DOM 数量，不减轻单批数据量与依赖项导致的更新。 |
| **存储** | 单文件全量读写、`writeFileSync`；图片 Base64 放大体积；无索引，查删依赖数组扫描。 |
| **批量操作卡死** | 清理等对每条调用 `remove`/`removeCollect`，每条可能触发写盘与视图刷新；无统一批处理事务。 |
| **快捷键** | 已有 feature + layer，但 **核心业务** 仍在组件与 hook；自定义仅限 overrides，缺 **Action 目录 + UI 绑定**。 |

## 五、优化方向汇总（与 Cascade 稿对齐，略压缩）

1. **存储层**：分片文档 / utools.db 多 doc；大图外链；能异步则异步（注意 API 是否同步）。  
2. **渲染**：虚拟滚动；图片懒加载与缩略图；过滤结果缓存或 Web Worker 预筛（视数据量）。  
3. **批量**：批量 API（多 id 一次内存更新 + 一次持久化）、进度与可取消。  
4. **快捷键**：引入 **Action 层**（输入：`{ action, payload }`），hotkey 仅解析为 Action；与 `hotkeyRegistry` 的 feature 可做 id 映射表逐步迁移。
