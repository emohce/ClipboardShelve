# Clipboard 实施错漏笔记

## 背景

在按 `260404-codex-Clipboard修复计划.md` 落地第一轮修复时，出现了新的问题：

- 方向键移动一度完全失效

这说明第一轮实施里，虽然方向正确，但对当前代码实际运行链路的判断有偏差，导致收敛过猛。

## 本次实施中遇到的问题

### 1. 误判了方向键的真实主链路

原本判断是：

- `ArrowUp/ArrowDown` 主要由 `HotkeyProvider -> hotkeyRegistry -> list-nav-up/down` 驱动
- `unifiedKeyHandler` 是多余的第二套推进源

实际落地后发现：

- 当前环境下，方向移动的实际可用链路仍然高度依赖 `unifiedKeyHandler`
- 直接把这套逻辑收掉后，方向键出现了“完全失效”

这说明：

- 当前代码虽然存在双重驱动风险
- 但不能简单认为 hotkey registry 已经能完整替代 `unifiedKeyHandler`

### 2. 去重策略过于激进，没有先做最小验证

第一轮修改时，采取了：

- 直接收缩 `unifiedKeyHandler`
- 试图让方向键完全交给 hotkey feature

问题在于：

- 没有先在代码里确认两套链路谁是主、谁是备
- 没有先做“只屏蔽物理方向键在 hotkey feature 中的重复处理”这种更小的收敛

正确顺序应该是：

1. 先恢复可用性
2. 再做重复触发隔离
3. 最后再决定是否删除一整套链路

### 3. 对虚拟滚动 fallback 的处理方式不够干净

为了快速绕过“最多滚半屏”的问题，修改时用了插入提前 `break` 的方式，让逻辑先短路到直接滚目标位置。

这虽然暂时达到了“避免半屏限制”的目的，但留下了明显的脏代码痕迹：

- `if (false) {}`
- 后方存在不可达分支

这类代码不应该作为最终状态保留。

### 4. 实施时受文件局部内容和编码影响，补丁命中不稳定

在 `ClipItemList.vue` 中，多处中文注释存在编码异常，导致：

- `apply_patch` 多次无法稳定命中完整代码块
- 只能退化为更小块、逐行式补丁

这带来了两个风险：

- 修改效率下降
- 更容易留下局部脏状态

## 已确认的正确修正

以下修正是有效且应保留的：

1. 修复收藏页强制删除分支中 `currentActiveIndex` 未定义的问题
2. 删除父组件中“同步设置一次 activeIndex，再异步再设一次”的双重恢复
3. 将删除后的恢复收敛为一次主修正逻辑

这些点属于明确 bug 修复，不应该回退。

## 本次实施暴露出的工作原则

后续继续修这个问题时，应遵守以下约束：

1. 不要再次直接移除 `unifiedKeyHandler` 的方向键能力
2. 去重时优先做“物理方向键只由一侧消费”，而不是先删整条链
3. 任何关于导航主链路的调整，都要先做最小可用验证
4. 清理虚拟滚动 fallback 时要做完整替换，不要留下不可达代码
5. 如果 `apply_patch` 反复命中失败，要先缩小目标块，再安排一次专门清理

## 下一步建议

下一轮应聚焦于“精修”，不要再做大范围收敛：

1. 清理 `ClipItemList.vue` 里滚动 fallback 的脏分支
2. 保持方向键继续可用的前提下，消除重复推进
3. 继续验证删除后错位问题是否已经收敛
4. 再单独处理“某些元素无法步入”的问题

## 结论

这次实施的主要错误不是方向判断完全错，而是：

- 在没有充分验证当前真实运行链路前，过早把“重复链路”当成“可删除链路”

后续修复应从“保可用，再收敛”切换为“先确认主链路，再做最小化去重”。
