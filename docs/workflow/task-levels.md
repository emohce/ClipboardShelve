# Task Levels

## 目标
- 按任务复杂度决定需要的流程深度，避免所有任务都走同样重的流程。

## L1 小改动
- 适用：单文件、小范围、低风险、没有新架构决策。
- 最低要求：直接实现 + 简短验证 + 判断是否需要知识沉淀。
- 可跳过：完整 `spec`、完整 `tasks`。
- 示例：文案修正、局部样式微调、明确根因下的小修复。

## L2 中等改动
- 适用：涉及多个文件、需要明确调用链、存在一定回归风险，但边界相对清晰。
- 最低要求：`plan + implement + verify + knowledge-capture`。
- 通常需要：说明命中/未命中历史错误记录。
- 示例：组件交互调整、单模块逻辑重组、列表行为修正。

## L3 高风险改动
- 适用：涉及 `public/`、`src/global/`、持久化、热键链、窗口行为、跨层交互或架构取舍。
- 最低要求：`spec -> plan -> tasks -> implement -> verify -> knowledge-capture`。
- 必须补充：影响范围、回退点、必要时新增 `ADR`。
- 示例：插件入口脚本修改、快捷键链改造、窗口生命周期行为调整。

## 判断规则
- 若同时满足多个等级，按更高等级处理。
- 只要改动落在高风险目录，默认至少按 `L3` 处理。
- 若命中历史失败模式且仍存在多个未知量，至少按 `L2` 处理。
