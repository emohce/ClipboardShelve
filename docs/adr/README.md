# ADR Rules

## 目标
- 记录长期有效的技术决策、架构取舍和团队约束，避免同一问题反复重议。

## 何时新增 ADR
- 明确选择 A 而不是 B，且该选择会持续影响后续实现。
- 因运行环境、兼容性、稳定性、维护成本而放弃某条技术路径。
- 修改仓库级规则、实现边界或高风险模块处理方式。

## 何时不写 ADR
- 一次性小修复。
- 仅影响当前任务、不会复用的临时判断。
- 还没有证据支持的猜测。

## 命名规则
- 文件名格式：`YYYY-MM-DD-slug.md`
- 决策编号格式：`ADR-YYYY-MM-DD-slug`

## 最低结构
- 背景
- 决策
- 备选方案
- 影响
- 后续约束
