## 2. 当前项目整体结构

项目是一个基于 Vue 3 的 uTools 插件，运行结构不是普通 Web 页面，而是“前端界面 + preload 注入能力 + 本地监听程序 + 本地 JSON 数据库存储”的组合。

核心目录：

- `src/`
  - Vue 前端页面、组件、hooks、全局模块
- `public/`
  - uTools 插件入口资源
  - preload 脚本
  - 原生监听程序桥接文件
- `docs/`
  - 历史开发记录和说明文档
- `docs/VersionDesc/`
  - 当前正式使用的版本说明目录，存放基线文档、增量记录、全量功能记录

最重要的代码文件：

- `src/global/initPlugin.js`
- `src/views/Main.vue`
- `src/cpns/ClipItemList.vue`
- `src/hooks/useClipOperate.js`
- `src/views/Setting.vue`
- `src/global/hotkeyRegistry.js`
- `src/global/hotkeyBindings.js`
- `public/preload.js`
- `public/listener.js`

## 3. 应用启动链路

启动入口在 `src/main.js`：

1. 先执行 `initPlugin()`
2. 再创建 Vue 应用
3. 挂载 `App.vue`

这意味着插件的宿主能力、数据库、事件监听、窗口管理，都是在 Vue UI 启动前初始化的。

`App.vue` 负责顶层页面切换：

- `Main.vue`：主界面
- `Setting.vue`：设置页
- `HotkeyProvider.vue`：全局快捷键捕获与分发入口

当前顶层结构是“单页面双视图切换”，不是多路由模式。

## 4. 宿主桥接与运行环境

### 4.1 preload 注入

`public/preload.js` 把以下能力注入到了 `window.exports`：

- `utools`
- 文件系统能力：`existsSync`、`readFileSync`、`writeFileSync`、`mkdirSync`、`watch`
- 路径分隔符 `sep`
- `crypto`
- `listener`
- Electron 剪贴板能力：`clipboard`、`nativeImage`
- 时间模块 `time`
- `Buffer`

### 4.2 前端对宿主层的依赖方式

前端代码大量依赖：

- `window.exports`
- `window.db`
- `window.listener`
- `window.remove`

这说明当前架构不是严格分层依赖注入，而是“通过全局对象桥接宿主能力”。

### 4.3 设计影响

优点：

- 前端调用简单，改动成本低
- uTools 环境能力接入直接

代价：

- 全局状态较多
- 组件和宿主层耦合偏强
- 做复杂重构时要避免一次性打散全局对象，否则风险很高

## 5. 数据存储模型

当前不是 sqlite，而是本地 JSON 文件。

数据库核心类定义在 `src/global/initPlugin.js` 中的 `DB`。

默认数据库结构：

```json
{
  "data": [],
  "collects": [],
  "collectData": [],
  "tags": [],
  "tagUsage": {},
  "createTime": 0,
  "updateTime": 0
}
```

字段含义：

- `data`
  - 普通历史记录
- `collects`
  - 收藏项目 ID 列表
- `collectData`
  - 收藏项目完整数据
- `tags`
  - 全局标签列表
- `tagUsage`
  - 标签使用统计

## 6. 收藏模型的核心特点

当前收藏逻辑不是简单布尔标记，而是“普通历史”和“收藏数据”分离存储”。

### 6.1 收藏时

- 从 `data` 中找到目标项
- 深拷贝该项
- 从 `data` 中移除
- 加入 `collects`
- 插入 `collectData`
- 初始化：
  - `collectTime`
  - `tags`
  - `remark`

### 6.2 取消收藏时

- 从 `collects` 中删除 ID
- 从 `collectData` 找到完整数据
- 从 `collectData` 中移除
- 更新 `updateTime`
- 重新插入 `data` 头部

### 6.3 这套设计的直接影响

优点：

- 收藏数据可独立管理
- 收藏记录不受普通历史清理逻辑直接影响
- 标签、备注等收藏特有属性更容易单独维护

风险：

- 删除逻辑不能按“单数组删一条”思维处理
- 收藏和普通历史之间的迁移一致性很重要
- 同一个 `id` 在不同阶段只应该存在于一个主集合中，否则会出现 UI 状态和操作错乱

后续任何涉及收藏、删除、批量清理、搜索的需求，都必须先确认是否会破坏这个约束。

## 7. 数据迁移与兼容逻辑

`DB.init()` 中存在多段迁移逻辑，说明当前版本需要兼容历史数据格式。

已识别的迁移/规范化逻辑包括：

- 旧 `item.collect` 迁移到 `collects`
- `collectData` 缺失时从历史数据回填
- `collectTime` 自动补齐
- `tags` 与 `remark` 自动补齐
- `locked` 自动补齐
- 文件类型的 `originPaths` 自动补齐
- 超时普通历史记录按 `maxage` 清理
- `collects` 和 `collectData` 做一致性修复

结论：

- 后续如果修改数据库结构，必须继续保持增量兼容
- 不建议随意删除现有迁移逻辑
- 若新增字段，建议延续当前“初始化时规范化补齐”的实现习惯

## 8. 剪贴板监听与入库链路

### 8.1 原生监听入口

`public/listener.js` 会根据系统平台启动外部监听程序：

- Windows：`clipboard-event-handler-win32.exe`
- Linux：`clipboard-event-handler-linux`
- macOS：`clipboard-event-handler-mac`

监听程序输出 `CLIPBOARD_CHANGE` 时，前端宿主层收到 `change` 事件。

### 8.2 入库流程

核心处理发生在 `src/global/initPlugin.js`：

1. 监听剪贴板变化
2. 读取当前剪贴板数据
3. 识别内容类型：
   - text
   - image
   - file
4. 生成标准 item
5. 调用 `db.addItem(item)` 写入普通历史
6. 通过监听事件推动视图刷新

### 8.3 失败回退

当前实现包含监听失败时回退轮询模式的逻辑。

说明：

- 当前系统对稳定性有考虑
- 监听层的问题可能表现为“偶发丢记录”“重复记录”“切换窗口时异常写入”
- 如果后续要优化监听性能或去重规则，应优先改 `initPlugin.js`，不要先在页面层做兜底补丁

## 10. Tab 与子 Tab 结构

`src/cpns/ClipSwitch.vue` 管理主 tab：

- `collect`
- `all`
- `text`
- `image`
- `file`

收藏 tab 还有子 tab：

- 固定项：`*全部*`
- 动态项：所有有使用量的标签

这些标签来自：

- `window.db.getTags()`
- `window.db.getTagUsage()`

排序规则是按使用频率降序。

这意味着：

- 收藏视图不只是“一个 tab”，而是“主 tab + 动态标签子 tab”
- 如果要优化收藏导航体验，优先看 `ClipSwitch.vue` 与 `Main.vue` 的联动

## 11. 列表展示与交互层

`src/cpns/ClipItemList.vue` 负责单条列表项的渲染和大部分列表交互。

处理内容包括：

- 当前活动项高亮
- 多选状态
- 收藏标记、锁定标记、标签展示
- 文本内容预览
- 图片预览

- 图片悬浮预览
- 文本长内容预览

这是一个较重的组件，说明当前列表层承载了较多职责。

后续如果出现以下类型需求，通常优先修改这里：

- 列表项显示优化

## 12. 操作行为层

单条记录的操作按钮由：

- `src/cpns/ClipOperate.vue`
- `src/hooks/useClipOperate.js`

共同完成。

当前支持的主要操作：

- 复制
- 查看详情
- 打开文件所在目录
- 打开来源路径
- 收藏
- 取消收藏
- 编辑标签
- 分词
- 保存到其他插件
- 删除
- 自定义功能

其中 `useClipOperate.js` 同时负责：

- 操作点击行为

这意味着后续如果你要：

- 增减按钮
- 修改按钮显隐规则
- 调整操作行为边界

优先改这里，不要先改列表模板。

## 13. 搜索规则

`Main.vue` 中搜索并不只是单字段模糊查询。

### 13.1 普通搜索

- 文本/文件按内容搜索
- 图片在有搜索词时通常被排除

### 13.2 星号搜索

当前支持 `*` 前缀收藏搜索模式：

- `*标签 关键词`
- `* 关键词`
- `*标签`

其含义大致为：

- `*` 表示优先基于收藏集合搜索
- `*` 后第一段可作为标签条件
- 后续可作为正文关键字

### 13.3 普通 tab 下的特殊行为

在非收藏 tab 下，如果输入是 `*` 前缀：

- 上方会展示“收藏结果块”
- 下方仍显示当前 tab 的普通结果

这是一种“混合搜索展示”模式。

优点：

- 收藏结果能快速浮现

代价：

- 逻辑较绕
- 用户心智不一定直观
- 后续做搜索体验优化时要先判断是保留还是简化这个模式

## 14. 多选与合并复制逻辑

`Main.vue` 中包含多选复制的完整逻辑。

规则概要：

- 如果只选中一条图片，直接复制图片
- 如果选中项里有图片或文件，按文件合并逻辑处理
- 如果全是文本，则按换行拼接为一个文本块
- 可以选择复制后立即粘贴
- 可以选择把合并结果持久化入库

这部分逻辑同时依赖：

- `ClipItemListRef` 的选中数据
- `utils/index.js` 的复制/粘贴工具
- `window.db.addItem`

后续如果优化多选体验，要重点确认：

- 合并后的数据格式是否符合原有系统
- 是否要保留“持久化合并结果”
- 图片/文件混合选择时的行为是否需要更精细

## 15. 删除、清理、锁定规则

这是当前系统最需要谨慎修改的一组规则。

### 15.1 普通删除

`DB.removeItemViaId(id, options)` 当前规则：

- 已收藏项目默认不允许删
- 锁定项目默认不允许删
- 只删普通历史 `data`
- `force` 可以绕过部分限制

### 15.2 收藏页删除

收藏页中不提供真正删除，语义上更接近“取消收藏”。

### 15.3 清理记录

`Main.vue` 提供时间范围清理：

- 1 小时内
- 5 小时内
- 8 小时内
- 24 小时内
- 7 天内
- 全部

并区分：

- 普通 tab 清理
- 收藏 tab 清理

收藏 tab 的“清理”本质上是批量取消收藏。

### 15.4 锁定机制

- 锁定项默认不参与删除/清理
- 需要强制操作时才允许绕过

结论：

- 后续任何“删除体验优化”都必须先定义清楚是在改：
  - 删除
  - 取消收藏
  - 清理
  - 强制删除
- 这四种语义目前不是一回事

## 16. 标签系统

标签主要服务于收藏项，不是普通历史通用字段。

当前已支持：

- 全局标签列表维护
- 标签使用次数统计
- 更新收藏项标签
- 更新收藏项备注
- 通过标签筛选收藏
- 标签建议
- 标签清理未使用项

相关逻辑主要在 `src/global/initPlugin.js` 和 `Main.vue` 的标签弹窗交互。

说明：

- 标签系统目前已经有一定完整性
- 后续更适合做“搜索体验、批量维护、排序展示、快捷入口”类优化
- 不建议轻易把标签扩展到普通历史，除非先确认数据模型要不要调整

## 17. 复制、粘贴与临时文件机制

公共工具在 `src/utils/index.js`。

关键能力：

- `copyAndPasteAndExit`
- `copyOnly`
- `createFile`
- 图片 data URL 与本地路径兼容处理

文件/图片处理特点：

- 图片可能是 base64 data URL，也可能是路径
- 文本或图片可转存临时文件
- 文件项内部数据是 JSON 字符串数组

这意味着：

- 后续做“复制失败”“图片兼容问题”“多平台路径问题”时，要先从 `utils/index.js` 和 `initPlugin.js` 下手
- 不建议在页面层重复写一套复制逻辑

## 18. 设置页能力

`src/views/Setting.vue` 当前提供三大类设置：

### 18.1 存储设置

- 数据库路径
- 最大历史条数
- 保存时间

### 18.2 快捷键列表

- 展示当前生效快捷键树

### 18.3 功能设置

- 控制主页展示哪些操作按钮
- 自定义操作项新增/编辑/删除
- 功能顺序拖拽排序

设置持久化写入：

- `utools.dbStorage.setItem('setting', payload)`

注意：

- 当前提示“保存成功 重启插件生效”，说明部分设置不是实时热更新
- 如果后续增加设置项，要先判断是否支持即时生效，避免 UI 表述和实际行为不一致

## 19. 快捷键系统

快捷键系统是当前项目第二核心层。

### 19.1 组成

- `src/global/hotkeyRegistry.js`
  - 特性注册与统一分发
- `src/global/hotkeyBindings.js`
  - 快捷键与 feature 的绑定表
- `src/global/hotkeyLayers.js`
  - 图层栈管理
- `src/cpns/HotkeyProvider.vue`
  - 捕获 `keydown` 并调用分发器

### 19.2 机制

1. `HotkeyProvider` 捕获 `keydown`
2. `dispatch()` 根据当前 layer 和 state 查找匹配 binding
3. 找到 feature 列表后逐个执行 handler
4. handler 可返回 handled 配置决定是否阻止默认行为

### 19.3 当前 layer

- `main`
- `setting`
- `clear-dialog`
- `clip-drawer`
- `full-data-overlay`
- `tag-search`
- `tag-edit`

### 19.4 当前主状态

`main` 下至少有这些 state：

- `normal`
- `search`
- `multi-select`

### 19.5 设计结论

优点：

- 键盘行为统一
- 支持图层隔离
- 支持设置覆盖绑定

风险：

- 全局捕获，容易影响输入框与弹窗
- 快捷键问题往往不是单文件问题，而是 layer/state 选择错误

后续凡是“某个键行为不对”的需求，建议按这个顺序排查：

1. 当前 layer 对不对
2. 当前 main state 对不对
3. binding 是否匹配
4. feature handler 是否提前消费事件

## 20. 当前代码的主要耦合点

这是后续迭代最值得提前注意的部分。

### 20.1 `initPlugin.js` 过重

同时承载：

- 数据库模型
- 数据迁移
- 剪贴板解析
- 监听回退
- 视图刷新事件
- 宿主桥接初始化

风险：

- 小改动容易牵动多处行为

建议：

- 短期内继续最小改动
- 中期如要稳定演进，可逐步拆出：
  - db 模块
  - clipboard parser 模块
  - listener manager 模块

### 20.2 `Main.vue` 过重

同时负责：

- 状态编排
- 过滤组装
- 批量操作
- 弹窗行为
- 快捷键注册

风险：

- 主页面需求容易造成持续膨胀

建议：

- 当前版本继续按功能点做局部修正
- 如果后续优化较多，可考虑抽离：
  - 搜索过滤逻辑
  - 清理逻辑
  - 标签弹窗联动逻辑

### 20.3 `ClipItemList.vue` 过重

列表层已经承担了：

- 展示
- 交互
- 预览
- 抽屉
- 快捷键行为

建议：

- 后续如果大量改列表行为，优先做局部函数拆分，不建议直接大拆组件

## 21. 后续迭代建议方法

后续讨论需求时，建议按下面格式推进：

### 21.1 先定义目标

明确是以下哪一类：

- 修 bug
- 优化交互
- 优化性能
- 调整数据规则
- 新增功能

### 21.2 再定义边界

至少明确：

- 影响哪个 tab
- 是否影响收藏
- 是否影响搜索
- 是否影响快捷键
- 是否影响已有数据兼容

### 21.3 再选方案

方案评估优先顺序建议：

1. 最小改动且不破坏现有数据模型
2. 行为更一致，减少歧义
3. 能被当前快捷键系统和设置系统承接
4. 必要时再考虑结构性调整
