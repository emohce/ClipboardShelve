# Implementation Constraints

## 目标
- 在实现阶段保持最小改动、风格一致、闭环完整。

## 通用约束
- 先读相关文件，再动手修改。
- 优先修根因，不做表面绕过。
- 非必要不新增依赖、不改目录结构、不改公共命名。
- 不擅自改动无关文件。
- 修改必须覆盖边界条件、空值、异常路径、异步行为。

## 当前仓库实现约束
- 使用现有 Vue 3 + JavaScript 代码风格，不引入 TypeScript。
- 保持 2 空格缩进与现有 SFC 结构。
- 组件命名沿用现有约定，避免为了“更优雅”重命名。
- 复用现有 `src/global/`、`src/utils/`、`src/hooks/` 能力，避免重复造轮子。
- Less 样式优先在已有文件增量调整，避免新增无必要样式入口。
- `dist/` 为构建产物，不手动编辑。

## 实施顺序
- 先改底层约束与数据流，再改界面呈现与交互细节。
- 先保证行为正确，再补充文档或说明。
- 如果计划包含多个任务，只实现当前指定任务，不扩写未批准范围。

## 风险控制
- 涉及 `public/plugin.json`、`public/preload.js`、`public/listener.js` 时，必须明确说明影响范围。
- 涉及持久化、快捷键、剪贴板监听、窗口行为时，优先保守处理。
- 如发现需求与现有实现冲突，先记录冲突点，再调整方案，不静默覆盖旧逻辑。

## 完成标准
- 代码能构建。
- 行为符合 Spec 与 Plan。
- 相关文档和验证记录同步更新。
