# Spec Extraction Rules

## 目标
- 在编码前把问题定义写清，形成 `01-spec.md`。
- Spec 只定义做什么、为什么做、完成标准，不提前展开实现细节。

## 输入要求
- 用户原始需求
- 相关背景文档或 issue
- 当前代码库中已存在的约束、入口、配置、命名方式

## 输出结构
```md
# Feature: 名称

## 1. 目标
## 2. 背景
## 3. 用户场景 / 使用场景
## 4. 非目标
## 5. 验收标准
## 6. 边界与异常
## 7. 影响范围
## 8. 待确认项
```

## 编写规则
- 目标必须是明确行为变化，不写模糊愿景。
- 背景说明当前问题和触发原因，不重复目标。
- 用户场景按真实流程拆开，覆盖成功、失败、空值、异常路径。
- 非目标用于收敛范围，避免 AI 扩写无关能力。
- 验收标准必须可验证，优先写成行为、数据或性能约束。
- 边界与异常必须覆盖并发、空数据、兼容性、回退路径。
- 影响范围至少点出模块、页面、组件、脚本、配置、文档中的相关部分。

## Spec 不应包含
- 具体实现方案
- 详细文件改动步骤
- 任务拆分
- “后续再看”“暂不确定但先做”这类逃避性表述

## 质量门槛
- 读完 Spec 后，另一个工程师应能准确复述需求边界。
- Spec 必须能直接作为 Plan 输入，不需要再翻聊天记录补全上下文。
