# Verification Checklist

## 目标
- 每次实现后形成真实可追踪的 `04-verify.md`。
- 验证优先基于当前仓库可执行命令和最小人工回归路径。

## 默认验证顺序
1. 代码静态检查
2. 构建验证
3. 关键路径人工验证
4. 文档与记录回填

## 当前仓库验证基线
- 安装依赖：`npm install`
- 本地开发：`npm run serve`
- 生产构建：`npm run build`
- 当前仓库未配置 lint/test 脚本，不虚构执行结果

## `04-verify.md` 建议结构
```md
# Verify

## T1 任务名称
- status:
- build:
- manual-check:
- docs:
- notes:
```

## 记录规则
- `status` 只写 `pass`、`partial`、`fail`。
- `build` 明确写执行命令与结果。
- `manual-check` 写最小可复现验证步骤与观察结果。
- `docs` 写是否更新说明文档、版本说明、规则文档。
- `notes` 只记录未覆盖项、环境限制、风险残留。

## UI/插件专项检查
- 页面改动至少说明入口页面和主要交互是否正常。
- 样式改动至少说明桌面窗口尺寸下是否存在遮挡、溢出、错位。
- 涉及 uTools 插件能力时，说明是否只做网页构建验证，还是已在插件环境验证。
- 涉及构建产物时，确认 `dist/plugin.json` 是否存在。

## 禁止项
- 没跑就写通过。
- 用“理论上可行”代替验证结果。
- 把待办项写成已完成结果。
