# AI Writing Style

## 目标
- 输出必须服务于当前工程任务，优先帮助实现、验证、交付。
- 结论先行，再给必要依据，不写空泛分析。

## 语言与格式
- 默认使用简体中文回答。
- 代码、命令、提交信息保持原语言语境。
- 优先使用短段落或单层要点列表，不使用嵌套列表。
- 避免寒暄、重复背景、无结论铺垫。

## 文档风格
- 默认使用简体中文回答。
- 文档标题直接写任务对象，不写口号式标题。
- 每份文档只承载一个主目的，避免 spec、plan、verify 混写。
- 使用明确可验证表述，避免“优化体验”“增强稳定性”这类空描述。
- 引用代码或文件时，优先给出明确路径, 提供相对目录带行号的mdlink。

## 工程表达要求
- 代码注释遵循 `04-implementation-constraints.md` 中的「简体 + 英文术语混排」规范（说明用中文，标识符与 API 名保持英文）。
- 需求要写清目标、范围、验收标准、边界条件。
- 方案要写清受影响文件、调用链、数据结构、接口变化、风险。
- 任务要可独立实现、可独立验证、可独立回滚。
- 验证记录只写真实执行结果，不写主观判断。

## 禁止项
- 不把猜测写成事实。
- 不省略关键限制条件。
- 不输出无法执行的“建议清单式废话”。
- 不把一次性会话内容当作长期规则。
