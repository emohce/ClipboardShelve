# Task Decomposition Rules

## 目标
- 基于 `02-plan.md` 生成 `03-tasks.md`。
- 每个任务必须原子化，便于逐项实现、验证、回写。

## 输出格式
```md
# Tasks

- [ ] T1 任务名称
- [ ] T2 任务名称
- [ ] T3 任务名称
```

## 拆分规则
- 一个任务只解决一个明确目标。
- 一个任务完成后应有明确验证动作。
- 任务顺序要符合依赖链，先基础约束，后 UI/交互，最后文档与验证。
- 大改动按“数据层 / 业务层 / UI 层 / 验证层”拆分。
- 涉及多个目录时，优先按闭环而不是按文件数量拆分。

## 单个任务描述要求
- 任务名要能直接看出动作和对象。
- 如有必要，在任务下补一句预期结果，但保持精简。
- 不写“完善细节”“处理相关逻辑”这种模糊描述。

## 推荐检查点
- 是否能单独提交。
- 是否能单独验证。
- 是否能在失败时局部回滚。
- 是否依赖未完成任务的隐式前提。

## 当前仓库专项建议
- UI 结构调整与样式调整尽量拆分，避免一次同时改 `vue` 和多个 `less` 造成回归难排查。
- `src/global/`、`public/`、`src/views/` 的跨层改动优先拆成独立任务。
- 文档、验证记录、版本说明更新放在最后单独任务。
