# Feature: 列表移动效果与删除状态修复

## 1. 目标
- 收敛主列表键盘导航行为，使单步移动、长按连续移动、翻页与边界滚动在同一套展示列表上表现一致，不再出现“可见项也强行滚动”“长按连续跳页错位”“删除后选中态漂移”。
- 修复删除、锁定、收藏图标渲染与选中态联动的错乱，确保删除后自动落点稳定、展示数据与持久化数据不混乱、列表不出现明显割裂或闪烁。
- 保持搜索、筛选、收藏结果混排与虚拟滚动场景下的高亮、可见性、性能一致性，列表常规渲染应满足需求文档提出的 `300ms` 内完成展示目标。

## 2. 背景
- 原始需求见 [../BjVkZ-zt-move.md](../BjVkZ-zt-move.md)。
- 原始需求明确指出“鼠标悬浮的选中效果很好，滚动也很好”，说明当前问题主要集中在键盘移动没有复用同等级别的可见性与高亮体验。
- 当前列表移动涉及多套触发路径：单步导航、长按连续导航、分页导航、边界加载更多、hover 高亮、删除后恢复。原始需求要求它们不能互相冲突、不能重复调用，同一时间只能有一套不冲突的底层移动逻辑运行。
- 当前用户已观察到以下实际问题：
  - 单步移动时，目标项可见仍发生滚动，或高亮落在隐藏范围内导致“移动了但看不到当前在哪一行”。
  - 长按上下键未稳定复用现有“分页并居中”能力，而是出现逐项强行居中、抖动或跳页不一致。
  - 删除 item 后会出现选中效果错乱、移动错误、元素渲染错误、展示数据与持久化数据短暂不一致。
  - 锁状态与收藏状态更新不及时，行头图标可能发生异常换行并挤压主体内容。

## 3. 用户场景 / 使用场景
- 场景 1：用户单击 `ArrowUp` / `ArrowDown` 时，如果目标项已经完整可见，则仅更新键盘选中高亮，不触发额外滚动；如果目标项不可见或被裁切，则滚动到可见位置。
- 场景 2：用户长按 `ArrowUp` / `ArrowDown` 时，列表复用现有“分页并居中”的导航模型，连续移动保持顺滑、可预测，不应每一步都强行把单条 item 生硬居中。
- 场景 3：用户在顶部或底部边界继续长按时，列表应优雅停止，或在允许加载更多时按既有规则接续；不能出现重复触发、多套逻辑争抢、明显抖动、空滚动或越界高亮。
- 场景 4：用户删除当前高亮项后，默认自动选中下一条；若下一条不存在，则回落到新的最后一条；页面渲染不应出现割裂、闪烁、跳回错误位置。
- 场景 5：用户进行多选删除、强制删除、取消收藏或搜索结果删除后，展示层高亮、选择集合与底层数据更新必须一致，不能出现“删的是 A，亮的是 B”。
- 场景 6：用户切换锁定状态或收藏状态后，当前列表应立即反映状态变化；锁/收藏图标在宽度紧张时需要自动缩略展示，不应异常换行或把主体内容撑乱。
- 场景 7：用户在搜索、锁筛选、`*` 收藏混合搜索或懒加载场景中继续键盘导航时，高亮与滚动规则仍然作用于实际展示结果，不因列表来源不同而错乱。

## 4. 非目标
- 不重做整个快捷键系统。
- 不替换现有虚拟滚动方案，不新增依赖。
- 不修改 [public/plugin.json](../../../public/plugin.json)、[public/preload.js](../../../public/preload.js)、[public/listener.js](../../../public/listener.js)。
- 不改动 tab 体系、搜索语法、收藏数据模型和全局快捷键配置名。
- 不在本阶段引入新的性能监控埋点或设置项。

## 5. 验收标准
- 单步导航
  - 单击 `ArrowUp` / `ArrowDown` 只触发一次明确的高亮变更。
  - 目标项已完整可见时不滚动。
  - 目标项不可见或被裁切时，仅滚动到可见，不出现超额跳动。
- 长按导航
  - 长按 `ArrowUp` / `ArrowDown` 复用现有分页能力推进高亮，不再逐项强行居中。
  - 长按期间目标项居中且选中样式持续明确。
  - 一次长按不会再叠加第二套自动滚动或第二套索引修正。
- 删除与高亮恢复
  - 删除当前项后，优先选中原位置上的下一条；若越界则回落到新列表最后一条；空列表时高亮安全归零。
  - 多选删除、强制删除、取消收藏、搜索删除等路径完成后，高亮、选择集合、展示内容保持一致。
  - 删除不应造成明显闪烁、割裂或持久化数据与展示数据错位。
- 锁定与收藏状态
  - 切换锁定或收藏后，当前行状态立即刷新，不依赖再次移动或再次选中。
  - 图标、时间、标签在单行头部中稳定排布；空间不足时允许缩略，但不异常换行。
- 搜索与筛选一致性
  - 搜索、锁筛选、收藏混合搜索下的导航、删除恢复、选中态展示都基于当前展示列表。
  - 顶部收藏结果块存在时，不得破坏主列表索引、高亮或删除落点规则。
- 稳定性与性能
  - 列表移动、删除、锁定切换后，不引入新的运行时错误、虚拟滚动错位或明显性能退化。
  - 常规列表状态更新在用户可感知范围内应满足需求提出的 `300ms` 目标。

## 6. 边界与异常
- 空列表、仅一条数据、删除至 0 条时不抛错，不出现非法索引。
- 长按到底部时，如存在更多数据可加载，则进入加载后的后续移动；如没有更多数据，则停在最后一条并保持选中可见。顶部同理。
- 搜索、筛选、收藏 tab、普通 tab、锁定筛选与多选状态切换过程中，移动与删除逻辑不能把当前选中项带到隐藏范围而失去可见反馈。
- `collect` 标签页继续保持现有删除语义约束，不在本次 spec 中扩写为新的破坏性删除规则。
- 锁定项默认删除受限，强制删除例外；本次只修复状态同步与展示，不改变安全默认值。
- 由于收藏与普通历史分离存储，删除与取消收藏后的展示恢复必须避免把收藏项与普通项混淆。

## 8. 待确认项
- “分页并居中”默认以仓库中现有分页能力为准，不额外定义新的分页步长。
- “300ms” 先按“主界面列表状态变更后的可感知完成时间”理解；verify 阶段需明确采用的实际检查口径。
- “借鉴鼠标悬浮效果”当前理解为复用其可见性与高亮体验，不要求键盘移动直接绑定 hover 事件本身。
