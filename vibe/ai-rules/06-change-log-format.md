# Change Log Format

## 目标
- 统一任务完成后的变更摘要、交付说明与知识沉淀说明。

## 最终输出必须包含
- 变更摘要
- 关键文件
- 风险 / 兼容性影响
- 验证状态
- 知识沉淀状态

## 推荐模板
```md
# 变更摘要
- ...

# 关键文件
- [path/to/file#L1](path/to/file#L1): 作用

# 风险 / 兼容性影响
- ...

# 验证状态
- 已执行: ...
- 未执行: ...

# 知识沉淀状态
- 命中历史记录: ...
- 新增 / 更新 Error Memory: ...
- 新增 / 更新 ADR: ...
- 新增 / 更新 Glossary: ...
```

## 仓库级 `changelog.md`

- **路径**：[changelog.md](../../changelog.md)（仓库根目录）。
- **排序（必须）**：**最新一轮更新永远写在文件最前面**。每轮新增一个 `## YYYY-MM-DD — 简述` 区块时，插在紧接文件内「排序说明」段落之后，**原有日期区块整体下移**；禁止把新内容追加在文件末尾。
- **建议结构**：每个日期区块内仍包含「变更摘要、关键文件、风险/兼容性、验证状态、知识沉淀状态」，可与上文「推荐模板」对齐或表格化，保持可读即可。

## 仓库级 `publishLog.md`（发布说明 / 用户视角）

- **路径**：[publishLog.md](../../publishLog.md)（仓库根目录，与 `changelog.md` 同级）。
- **与 `changelog.md` 的配对关系（必须）**：凡向 `changelog.md` 增加**一轮**含代码改动的详细说明时，**同一交付批次**须同步更新 `publishLog.md`。内部文档-only（仅 spec / EM / 规则、无用户可感知行为变化）可只在 `changelog` 记一笔，**但**若对外声称发版，仍建议在 `publishLog` 用最短一句说明「无用户可见变更」以免误解。
- **读者与语气**：写给**最终用户**与**发布稿**摘用；用通俗中文短句，避免开发者口吻。
- **内容边界（必须）**：
  - **只写用户能直接感知**的变化：界面、交互、性能体感、数据可靠性、插件入口行为等。
  - **禁止**：文件路径、类名、函数名、提交号、EM/ADR 编号、Speckit 流程、实现细节、内部重构（除非用户可见）。
- **篇幅（必须）**：**严格精简**。每条 `## 日期` 下建议 **1～6 条**短列表（必要时单条合并表述），单条宜一行；不写子章节、不写长段落。
- **排序（必须）**：与 `changelog.md` 相同，**最新日期区块永远在文件最前**；禁止在文末追加新轮次。
- **日期与版本（建议）**：标题使用 `## YYYY-MM-DD`，必要时在同一行或下一行括号注明 `vX.Y.Z` 或「uTools 插件市场说明用语」，与真实发版标签对齐。
- **验收**：发布前通读一遍，假设「完全不懂代码的用户」能否理解「这次多了什么、少了什么坑」。

## 规则
- 变更摘要只写本次真实改动。
- 关键文件优先列核心文件，不罗列所有附带修改。
- 风险部分必须说明未验证项、兼容边界、环境限制。
- 如果命中了历史错误记录，必须明确记录编号和采用 / 规避策略。
- 如果本次没有新增知识沉淀，也要明确写“无”。
